# UX Audit — Hersa Frontend
**Fecha:** 2026-05-01
**Autor:** Laura (UX Designer agent)
**Alcance:** Frontend React 19 en `/frontend/src/` — flujos, navegación, jerarquía de información, friction points y escalabilidad estructural.
**No incluido:** Decisiones de color, tipografía, o tokens de marca (dominio de UI Designer).

---

## 1. Inventario de flujos actuales

### Módulos y pantallas existentes

| Ruta | Módulo | Pantalla | Permiso requerido | Rol típico |
|------|--------|----------|-------------------|------------|
| `/login` | auth | Login form | público | Todos los roles internos |
| `/forgot-password` | profile | Solicitar reset de contraseña | público | Todos |
| `/reset-password?uid=&token=` | profile | Formulario reset contraseña | público (link por email) | Todos |
| `/profile` | profile | Ver datos de usuario + cambiar contraseña | autenticado (sin permiso especial) | Todos |
| `/tienda` | tienda | Placeholder "Próximamente" — 4 items futuros | `modules.access_tienda` | Comercial, Cajero |
| `/grados` | grados | Placeholder "Próximamente" — 4 items futuros | `modules.access_programador` | Comercial, Coordinador, Fotógrafo |
| `/admin` | admin | Dashboard con stats estáticos + actividad hardcodeada | `modules.access_admin` | Administrador, Gerente |
| `/` | — | Redirect automático a `/profile` | autenticado | — |

### Flujos de navegación documentados

**Flujo 1 — Login:**
`/login` → credenciales → POST → éxito → redirect `/` → redirect `/profile`

**Flujo 2 — Recuperación de contraseña:**
`/login` → "¿Olvidaste tu contraseña?" → `/forgot-password` → submit email/usuario → pantalla confirmación (estado success en el mismo componente) → [email] → `/reset-password?uid=&token=` → nuevo password → éxito (Snackbar/Alert) → link manual a login

**Flujo 3 — Navegación post-login:**
`/profile` (destino post-login) → sidebar → `/tienda` | `/grados` | `/admin` | `/profile`

**Flujo 4 — Módulos futuros (Tienda y Grados):**
`/tienda` o `/grados` → grilla "Próximamente" con 4 cards → dead end (no hay acciones)

## 2. Mapa de navegación e información jerárquica

### Estructura de navegación actual

```
AppHeader (fixed, 100% ancho menos sidebar en desktop)
  └─ Hamburger [mobile only] → abre NavSidebar como Drawer temporal
  └─ LogoutIcon [siempre visible]

NavSidebar (permanent drawer 240px en desktop; temporal en mobile)
  ├─ Brand title "EVENTOS HERSA"
  ├─ [Tienda]     → /tienda   [si modules.access_tienda]
  ├─ [Grados]     → /grados   [si modules.access_programador]
  ├─ [Admin]      → /admin    [si modules.access_admin]
  └─ [Mi perfil]  → /profile  [siempre visible]
  └─ Username display (read-only, bottom)
```

### Análisis de jerarquía por pantalla

**Login (`/login`):**
- Nivel 1: Título "Eventos Hersa" (orientador de marca)
- Nivel 2: Campos usuario + contraseña
- Nivel 3: Botón CTA "Iniciar sesión"
- Nivel 4: Link "¿Olvidaste tu contraseña?" (acción secundaria)
- Problema: El título es solo texto — no hay logo visible.

**Admin (`/admin`):**
- Nivel 1: Título "Administración" + subtítulo
- Nivel 2: Botones de acción (Exportar reporte, Nuevo usuario) — en la misma línea del título
- Nivel 3: 4 StatCards con métricas clave
- Nivel 4: Lista "Actividad reciente"
- Problema: Las acciones (Exportar, Nuevo usuario) compiten visualmente con el título en la misma fila. Los datos de los StatCards son hardcodeados (no vienen de la API).

**Perfil (`/profile`):**
- Nivel 1: Título "Mi perfil"
- Nivel 2: Card "Cambiar contraseña" (izquierda) + Card "Mi información" (derecha)
- Problema: En mobile, "Cambiar contraseña" aparece ANTES que "Mi información". El orden debería ser inverso: primero ver quién eres, luego la acción de cambiar contraseña.

**Tienda y Grados:**
- Solo hay una grilla de 4 cards "Próximamente" sin ninguna acción disponible.
- No hay fecha estimada ni contexto de qué hacer mientras tanto.

## 3. Análisis de fricción

### [FRICCIÓN ALTA] — Login no tiene logo Hersa

**Pantalla:** `/login`
**Descripción:** El formulario de login muestra solo el texto "Eventos Hersa" como h5. No hay logo. Para un usuario que accede por primera vez (ej. una coordinadora de colegio recibiendo acceso por primera vez), la ausencia de logo genera desconfianza sobre si está en el sitio correcto.
**Impacto:** Primera impresión de la plataforma — momento crítico de adopción.
**Resolución propuesta:** Agregar el logo Hersa (ya existe en assets) como elemento prominente sobre el formulario.

### [FRICCIÓN ALTA] — Post-login siempre aterriza en "Mi perfil"

**Pantalla:** `/` → redirect a `/profile`
**Descripción:** Todo usuario autenticado —independientemente de su rol— aterriza en "Mi perfil" después del login. Para el Gerente o el Administrador, lo primero que ven es un formulario de cambio de contraseña. No hay un dashboard contextual.
**Impacto:** El usuario no sabe qué hacer ni dónde está el trabajo real. Requiere navegación adicional para llegar a su módulo de trabajo.
**Resolución propuesta:** Implementar redirect post-login basado en rol: Administrador → `/admin`; Comercial/Programador → `/grados`; sin rol preferente → `/profile`.

### [FRICCIÓN ALTA] — Módulos "Próximamente" sin fecha ni contexto

**Pantalla:** `/tienda`, `/grados`
**Descripción:** Los módulos de Tienda y Grados muestran una grilla de 4 cards con "Próximamente" como texto. No hay ninguna acción disponible, ni fecha estimada, ni texto explicativo de qué podrán hacer cuando el módulo esté listo. El usuario llega, ve esto y no sabe si es un error, si necesita permisos adicionales, o simplemente si está todo bien.
**Impacto:** Confusión y pérdida de confianza. El usuario puede pensar que no tiene acceso y contactar soporte.
**Resolución propuesta:** Agregar un copy contextual ("Estamos trabajando en esta funcionalidad, estará disponible pronto.") y opcionalmente un estado de progreso o fecha estimada. También verificar: ¿debería el sidebar mostrar estos items si el módulo no está funcional?

### [FRICCIÓN ALTA] — Admin con datos hardcodeados

**Pantalla:** `/admin`
**Descripción:** Los 4 StatCards (usuarios activos, colegios registrados, ceremonias, ingresos) y toda la lista de "Actividad reciente" son strings hardcodeados en el archivo `ui.ts`. No se consumen de la API. Un administrador que usa este panel no está viendo datos reales.
**Impacto:** El administrador toma decisiones con datos falsos o confunde el estado del sistema. Es una fricción de confianza sistémica.
**Resolución propuesta:** Conectar a endpoints reales o mostrar claramente que es un mockup en desarrollo (ej. con un badge "Demo" visible). Esta es la fricción de mayor impacto negativo en confianza.

### [FRICCIÓN MEDIA] — Orden de cards en Mi Perfil en mobile

**Pantalla:** `/profile` en mobile (xs)
**Descripción:** El layout usa Grid2 con md:6 / xs:12. En mobile, la card "Cambiar contraseña" (columna izquierda del desktop) aparece arriba de "Mi información". Un usuario nuevo que llega a Perfil ve primero el formulario de cambio de contraseña, antes de ver sus datos propios.
**Impacto:** Desorientación en mobile. El usuario quiere ver quién está logueado, no cambiar la contraseña de entrada.
**Resolución propuesta:** Invertir el orden de los Grid2 children, o usar `order` de CSS para cambiar el orden en mobile.

### [FRICCIÓN MEDIA] — Logout sin confirmación

**Pantalla:** AppHeader (persistente en toda la app)
**Descripción:** El ícono de logout está en el AppHeader como botón de ícono sin confirmación. Un toque accidental (especialmente en mobile) ejecuta el logout inmediatamente, perdiendo el contexto de trabajo actual.
**Impacto:** En flujos de trabajo del día de toma fotográfica o ceremonia, un logout accidental interrumpe operaciones críticas (cajero procesando pago, fotógrafo verificando autorización).
**Resolución propuesta:** Agregar un Dialog de confirmación ("¿Seguro que quieres cerrar sesión?") antes de ejecutar el logout.

### [FRICCIÓN MEDIA] — Reset password: redirect post-éxito es manual

**Pantalla:** `/reset-password`
**Descripción:** Después de resetear la contraseña exitosamente, el usuario ve un mensaje de éxito pero debe hacer clic manualmente en el link "Volver al inicio de sesión". No hay redirect automático.
**Impacto:** Fricción menor pero innecesaria. El usuario está motivado a iniciar sesión y tiene que dar un paso extra.
**Resolución propuesta:** Agregar un auto-redirect a `/login` después de 3-5 segundos post-éxito, con countdown visible.

### [FRICCIÓN MEDIA] — ModuleGuard renderiza null en isLoading

**Pantalla:** Cualquier módulo protegido durante carga de permisos
**Descripción:** `ModuleGuard` retorna `null` mientras `isLoading === true`. El usuario ve pantalla en blanco breve mientras se cargan los permisos. No hay Skeleton ni indicador de carga.
**Impacto:** Parpadeo (flash) de contenido o pantalla vacía que puede percibirse como error.
**Resolución propuesta:** Reemplazar `return null` con un Skeleton o spinner apropiado para el tamaño del módulo.

### [FRICCIÓN MEDIA] — Sidebar no diferencia roles visualmente

**Pantalla:** NavSidebar
**Descripción:** El sidebar muestra u oculta items basándose en permisos, pero no hay ninguna indicación de qué rol tiene el usuario actualmente. Un usuario con múltiples roles no sabe desde qué "perspectiva" está operando.
**Impacto:** Confusión para usuarios con múltiples roles (ej. Gerente que también puede operar como Administrador).
**Resolución propuesta:** Mostrar el rol activo del usuario en la sección inferior del sidebar (donde hoy solo aparece el username).

### [FRICCIÓN MEDIA] — AuthGuard renderiza div vacío en isLoading

**Pantalla:** Toda la app durante verificación de auth
**Descripción:** `AuthGuard` retorna `<div className={styles.root} />` (vacío) mientras `isLoading`. No hay indicador de carga.
**Impacto:** Pantalla en blanco al iniciar la app o al hacer refresh con sesión activa. Puede percibirse como error de carga.
**Resolución propuesta:** Mostrar un spinner o pantalla de splash con el logo durante la verificación de auth.

## 4. Escalabilidad UX

### Estado actual del sistema de navegación

El sidebar tiene actualmente **4 items navegables** (Tienda, Grados, Admin, Mi Perfil). El sistema de permisos (`ModulePermission`) define 3 módulos protegidos (`access_tienda`, `access_programador`, `access_admin`) más el perfil sin restricción.

### Capacidad de escala del sidebar

El sidebar es un `List` vertical de MUI con items de altura fija (~48px por item) más el toolbar de marca (~64px), un divider, y la sección de usuario inferior (~52px). Total actual: ~300px de altura.

**Límites identificados:**

| Items en sidebar | Altura aproximada | Estado |
|-----------------|-------------------|--------|
| 4 (actual) | ~300px | Sin problema |
| 7 | ~444px | Sin problema en pantallas normales |
| 9 | ~540px | Conflicto potencial en monitores <768px de alto |
| 11+ | ~600px+ | Overflow garantizado en laptops estándar (768px alto) — requiere scroll o agrupación |

**El limite concreto es 8-9 items** antes de que el sidebar necesite overflow scroll en monitores de 768px de alto, que es común en laptops de 13-14 pulgadas. Sin scroll visible en el sidebar, los items del final quedarían ocultos sin ninguna señal visual.

### Ausencia de agrupación semántica

Los módulos futuros según el dominio incluyen al menos: Gestión de Promociones, Toma Fotográfica, Prom/Actividades, Día de Grado, Cajero/Pagos, Logística, Candidatos/Personal, Cotizador, Reportes, Configuración. Eso son 10+ módulos potenciales que hoy encajarían como items planos en el sidebar.

**El patrón flat-list del sidebar actual NO escala a 10+ módulos.** Sin agrupación, el sidebar se convierte en una lista larga sin jerarquía, y el usuario no puede entender rápidamente en qué área está trabajando.

### Header en mobile

El AppHeader en mobile muestra: [Hamburger] [Spacer] [Logout]. Al agregar más módulos, el header actual no crecerá — la navegación sigue en el drawer. No hay riesgo de ruptura del header por más módulos.

Sin embargo, si se agregan acciones globales al header (notificaciones, búsqueda global, switcher de contexto de Promoción), el header puede llenarse rápidamente. La implementación actual solo tiene 2 elementos funcionales y no reserva espacio para expansión.

### El permiso `modules.access_programador` es ambiguo

El permiso que controla el módulo "Grados" se llama `access_programador`. Este nombre no describe correctamente lo que protege. A medida que se agreguen más módulos, el sistema de permisos necesitará nombres semánticos consistentes. Si "programador" se refiere al rol interno "Planillador/Cajero", es incorrecto nombrarlo así; si se refiere a "programación de ceremonias", el nombre debería ser `access_grados` o `access_ceremonias`.

### Routing plano sin sub-rutas

Todas las rutas son de primer nivel (`/tienda`, `/grados`, `/admin`). Para módulos complejos como Grados (que tendrá Promociones, Grupos, Pagos, Logística, etc.), no existe ninguna estructura de sub-routing. Cuando se implemente la funcionalidad real, se necesitará routing anidado (`/grados/promociones/:id`, `/grados/grupos/:id/pagos`, etc.) que el router actual no tiene.

### Patrón "ComingSoonGrid" no escala como arquitectura de features

Los módulos Tienda y Grados usan `ComingSoonGrid` como página principal. Este componente es un dead end UX. Cuando llegue el momento de implementar las sub-features, se necesitará decidir si la "página principal del módulo" se convierte en un dashboard con sub-navegación (Tabs, segunda barra lateral, cards navegables) o en un redirect a la primera feature real. Esta decisión de arquitectura UX aún no está definida.

## 5. Recomendaciones priorizadas

### Alta prioridad — alto impacto, bajo esfuerzo

**P1 — Datos reales en Admin Dashboard**
Conectar los StatCards a endpoints reales del backend o, como paso intermedio, agregar un badge/nota visible "Datos de demostración" mientras los endpoints no existen. El riesgo de operar con datos falsos en producción es el mayor problema UX y de confianza del sistema actual.
Esfuerzo: bajo a medio (query RQ + cambio de componente).

**P2 — Logo en pantalla de Login**
Agregar el logo Hersa (activo ya existente en `src/shared/assets/`) sobre el formulario de login. Es una intervención de 10 minutos que elimina la mayor fricción de primera impresión.
Esfuerzo: muy bajo (agregar `<img>` o `<Box component="img">` en AuthModal).

**P3 — Redirect post-login basado en rol**
Cambiar el redirect de `/` a una función que evalúe los permisos y dirija al módulo principal del usuario. Administrador → `/admin`; usuario con `access_programador` → `/grados`; fallback → `/profile`.
Esfuerzo: bajo (lógica en `AuthGuard` o en la ruta `/`).

### Prioridad media — impacto significativo, esfuerzo moderado

**P4 — Confirmación de logout**
Envolver la acción de logout en un Dialog de confirmación. Protege operaciones críticas en campo (cajeros, coordinadores de grado).
Esfuerzo: bajo (Dialog + estado en AppHeader).

**P5 — Definir arquitectura de navegación intra-módulo antes de implementar features**
Antes de construir las sub-features de Tienda o Grados, definir si el patrón será Tabs (sub-navegación horizontal dentro del módulo), sidebar secundaria, o cards navegables como dashboard. Esta decisión tiene que tomarse UX-primero porque afecta la estructura de routing.
Esfuerzo: decisión de diseño sin costo de código. Entregable: documento de decisión o ux-spec por módulo.

**P6 — Skeleton en ModuleGuard y AuthGuard**
Reemplazar `null` y `<div />` vacíos con Skeletons contextuales. Elimina flashes de pantalla blanca.
Esfuerzo: bajo (2 cambios de componente).

**P7 — Invertir orden de cards en Perfil para mobile**
Poner "Mi información" primero y "Cambiar contraseña" segundo en mobile. Corrección de una línea con `order` CSS.
Esfuerzo: muy bajo.

### Baja prioridad — mejoras de calidad, pueden esperar

**P8 — Preparar sidebar para agrupación semántica**
Antes de que el sidebar llegue a 6+ items, refactorizar la lista de nav items para soportar grupos (`{ groupLabel, items[] }`). No requiere cambio visual inmediato, pero prepara la estructura de datos para cuando sea necesario colapsar grupos o agregar divisores semánticos.
Esfuerzo: medio (refactor de `NavSidebar` + `NAV_ITEMS`).

**P9 — Renombrar `access_programador` a un nombre semántico**
Alinear el nombre del permiso con el dominio real. Requiere coordinación backend.
Esfuerzo: bajo-medio (backend migration + frontend constants).

**P10 — Auto-redirect post reset-password**
Agregar countdown y redirect automático a `/login` después del éxito en `/reset-password`.
Esfuerzo: muy bajo.

## 6. Hallazgos sobre escalabilidad para módulos futuros

Los módulos documentados en `hersa-process.md` que aún no tienen representación en el frontend incluyen al menos:

- Gestión de Promociones (core del negocio)
- Cajero / Sesiones de cobro
- Toma fotográfica
- Logística / Asignación de personal
- Candidatos externos / Postulaciones
- Cotizador (pre-Promoción)
- Día de grado / QR scanning
- Solicitudes de reembolso
- Inconformidades / SLA
- Configuración del sistema

Cada uno de estos representa una sección de sidebar potencial. Con 10 módulos futuros más los 4 actuales, el sidebar llegará a 14 items — nivel donde la navegación plana y sin agrupación es inusable.

**El mayor riesgo UX a mediano plazo no es visual — es arquitectónico:**
el sistema de routing plano y el sidebar sin jerarquía no están preparados para el volumen de módulos que el dominio de negocio requiere. Esta decisión de arquitectura UX debe tomarse antes de que empiecen a implementarse los módulos de negocio reales.

## 8. Notas técnicas para el equipo

- El sistema de permisos (`ModulePermission` en `src/shared/types/permissions.ts`) ya tiene la infraestructura correcta para crecer. Añadir nuevos módulos es solo agregar un string al union type y una entrada en `NAV_ITEMS`.
- `ModuleLayout` es un componente sólido y reutilizable. Todas las páginas futuras deberían usarlo como wrapper.
- `ComingSoonGrid` cumplió su propósito de placeholder pero debe reemplazarse cuando los módulos sean funcionales — no extenderse como patrón de navegación.
- La estrategia de offline (ConnectivityIndicator + OfflineMutationDialog + MutationButton) está bien pensada para el contexto de campo (cajeros en días de toma fotográfica con conectividad variable). Es una fortaleza del sistema actual.
