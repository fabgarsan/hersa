# UI Audit — Hersa Frontend
**Fecha:** 2026-05-01
**Auditor:** Jose (ui-designer)
**Alcance:** `frontend/src/` — design system, componentes compartidos, módulos implementados, SCSS Modules, accesibilidad básica, escalabilidad UI.
**Rama auditada:** `main` (commit `23dcafb`)

---

## Resumen ejecutivo

El frontend de Hersa tiene una base de design system sólida: tokens centralizados, SCSS Modules aplicado con disciplina, y uso correcto del tema MUI. Los tres hallazgos más críticos para escalar son: (1) dos componentes de botón con responsabilidades solapadas que crearán confusión al agregar módulos, (2) ausencia de estados visuales explícitos en componentes clave (hover de tarjetas, focus rings, estados vacíos/error en tablas), y (3) el login no usa `AuthPageCard` — hay un patrón de pantalla de auth duplicado que divergirá al crecer.

## 1. Estado del Design System

### 1.1 Tokens de color

**Resultado: CORRECTO y centralizado.**

`frontend/src/shared/styles/theme.ts` y `frontend/src/shared/styles/_variables.scss` son la fuente única de verdad. Los valores coinciden exactamente con `brand-manual.md`. No se detectaron colores hardcodeados fuera de estos archivos en ningún componente auditado.

**Hallazgo menor:** `_variables.scss` no expone los colores semánticos `$success-light`, `$warning-light`, `$error-light`, `$info-light` (solo tiene los principales `$success-main`, etc.). Si un módulo futuro necesita usar el fondo claro de un estado semántico en SCSS, tendrá que hardcodear el valor o inventar un token propio. Recomendación: agregar los cuatro tokens light.

**Hallazgo menor:** `ModuleGuard.module.scss` usa la CSS variable MUI `var(--mui-palette-text-disabled)` para el ícono de acceso denegado porque ese token no existe en `_variables.scss`. Es una solución pragmática y correcta para este caso, pero documenta un gap: `$text-disabled` falta en el archivo de variables SCSS. Recomendación: agregar `$text-disabled: rgba(0, 0, 0, 0.38)` a `_variables.scss`.

### 1.2 Tokens tipográficos

**Resultado: CORRECTO en estructura, con gap de escala.**

`theme.ts` define correctamente la escala Inter + Playfair Display. La variante `display` (Playfair Display Bold, 3rem) está declarada en el tema con la augmentación TypeScript correspondiente. `textTransform: none` en botones está correctamente configurado.

**Hallazgo:** La escala tipográfica en `theme.ts` tiene `h1: fontSize: "2rem"` (32px), pero el brand manual especifica 32–40px. El valor está en el rango mínimo. No es un error, pero al agregar pantallas con más jerarquía visual podría sentirse pequeño. Sin urgencia.

**Hallazgo menor:** `NavSidebar.module.scss` usa `font-size: 0.9rem` (14.4px) para etiquetas de navegación hardcodeado en SCSS, en lugar de usar el token `body2` (0.875rem = 14px). Diferencia de 0.4px — no visible pero introduce un valor fuera de la escala.

### 1.3 Tokens de espaciado

**Resultado: EXCELENTE.**

La función `sp($n)` en `_variables.scss` es la implementación correcta del grid de 8px. Se usa consistentemente en todos los archivos SCSS auditados. No se detectaron valores de spacing hardcodeados fuera de la escala.

### 1.4 Border radius

**Resultado: GAP.**

`theme.ts` define `shape: { borderRadius: 8 }`. Sin embargo, `NavSidebar.module.scss` usa `border-radius: 4px` directamente en los nav items (`.navItem`, `.navItemActive`). Este es el único valor de border radius en toda la aplicación que no usa el token del tema.

**Impacto:** Inconsistencia visual: las tarjetas tendrán radio 8px por defecto de MUI, mientras los items de navegación activos tienen 4px. Al agregar más componentes con radio, puede amplificarse el drift.

**Recomendación:** Agregar `$border-radius-sm: 4px` y `$border-radius-md: 8px` a `_variables.scss`, o definir solo `$border-radius: 8px` y usar `v.sp(0.5)` para 4px.

### 1.5 Shadows

**Resultado: CORRECTO — sin shadows problemáticas.**

El design system Hersa define `elevation={0}` con border en lugar de sombras. Esto se respeta: `AuthModal` usa `elevation={0}` + `border: 1px solid rgba(0, 0, 0, 0.08)`. `StatCard`, `ComingSoonGrid`, `ProfilePage` usan `variant="outlined"`. No hay `elevation` mayor que 0 en componentes de contenido (solo el dialog usa la sombra default de MUI, lo que es correcto).

### 1.6 Configuración del tema MUI

**Resultado: CORRECTO y bien construido.**

- `ThemeProvider` + `CssBaseline` correctamente en `main.tsx`.
- `createCache({ prepend: true })` resuelve correctamente la batalla de especificidad SCSS vs Emotion.
- Todos los overrides de componente que existen están en el tema, no en archivos individuales.

**GAP:** El tema no tiene overrides de componentes MUI para establecer los comportamientos de marca de forma global:
- `MuiCard` / `MuiPaper`: no hay override global para `elevation={0}` como default — cada componente debe recordar pasarlo.
- `MuiButton`: no hay override global para `size="medium"` ni `disableElevation`.
- `MuiChip`: no hay override para asegurar que los Chips de estado usen las variantes semánticas correctas.

**Impacto en escalabilidad:** Sin estos overrides, cada módulo nuevo tendrá que recordar pasar `elevation={0}` a sus Cards, aumentando el riesgo de drift.

## 3. Consistencia Visual

### 3.1 Colores

**Resultado: CONSISTENTE en todos los módulos auditados.**

Ningún componente hardcodea colores fuera de `_variables.scss`. El único caso borderline es `NavSidebar.module.scss` que usa `rgba(201, 162, 39, 0.1)` y `rgba(201, 162, 39, 0.15)` para el fondo del item activo — estos son derivaciones del `$secondary-main` que no tienen tokens propios. Pragmáticamente correctos pero no tokennizados.

`StatCard.module.scss` usa `rgba(59, 109, 17, 0.08)` y `rgba(163, 45, 45, 0.08)` para chips de tendencia — misma situación: derivaciones de semánticos sin tokens.

**Recomendación de baja urgencia:** Agregar tokens de opacidad baja para estos backgrounds (`$success-bg-subtle`, `$error-bg-subtle`, etc.) cuando se creen más componentes que los usen.

### 3.2 Tipografía

**Resultado: MAYORMENTE CONSISTENTE, con una anomalía.**

La escala tipográfica Inter se respeta. `h5` se usa para títulos de página en `ProfilePage` y `AuthModal` (sin usar `ModuleLayout`). `h4`/`h6` usados para valores estadísticos y títulos de sección.

**Anomalía:** `ModuleLayout` usa `variant="h5"` para el título pero no está en la escala definida en `theme.ts` (que define h1–h4). MUI renderiza `h5` con sus valores default (no customizados en el tema). Esto significa que el título principal de todos los módulos (`ModuleLayout.title`) usa una variante no tokennizada.

**Recomendación:** Agregar `h5` y `h6` al tema Hersa con pesos y tamaños explícitos, o cambiar `ModuleLayout` a `variant="h4"` (que sí está en el tema).

### 3.3 Spacing

**Resultado: EXCELENTE.** Función `sp()` usada con consistencia en toda la base de código. Padding de contenido correctamente escalado: 16px (mobile) → 24px (tablet) → 32px (desktop) en `Layout.module.scss`.

### 3.4 Estados de componentes

**Las tarjetas (Paper/Card con `variant="outlined"`) no tienen hover state.** El digital-guidelines especifica hover con borde `primary.light` (`#1E3A5F`), pero esto no está implementado en ningún componente de tarjeta. Es el gap de estado más visible en la UI actual.

### 3.5 Iconografía

**Resultado: CORRECTO en librería, INCONSISTENTE en tamaños.**

Todos los íconos son de Material Icons (Google), disponible nativamente en MUI — correcto.

Tamaños encontrados:
- `StatCard`: `font-size: 28px` (hardcodeado en SCSS)
- `ComingSoonGrid iconWrapper`: `font-size: 40px` (hardcodeado en SCSS)
- `ModuleGuard accessDeniedIcon`: `font-size: 64px` (hardcodeado en SCSS)
- `ConnectivityIndicator icon`: `font-size: 16px` (hardcodeado en SCSS con `!important`)

El brand manual define 24×24px estándar en interfaces y 32×32px para énfasis. Solo `ComingSoonGrid` se acerca a estos valores. No hay tokens de tamaño de ícono definidos.

**Recomendación:** Agregar a `_variables.scss`:
```scss
$icon-sm: 16px;
$icon-md: 24px;
$icon-lg: 32px;
$icon-xl: 48px;
```

## 4. Accesibilidad Básica

### 4.1 Contraste de color

Todas las combinaciones de color principales auditadas cumplen WCAG AA (los ratios están verificados en `digital-guidelines.md §2`):
- Blanco sobre Marino `#0B1F3A`: 14.7:1 (pasa)
- Oro `#C9A227` sobre Marino `#0B1F3A`: 5.8:1 (pasa)
- Marino sobre Blanco: 14.7:1 (pasa)
- `#5F5E5A` sobre Blanco (`text.secondary`): 5.9:1 (pasa)

**ALERTA:** `rgba(255, 255, 255, 0.75)` (color de nav items inactivos en sidebar) sobre `#122640`. El valor efectivo del texto es aproximadamente `#C2C2C2` sobre `#122640`. Ratio calculado: ~7.2:1 — pasa AA. Sin embargo, el uso de opacidad en lugar de color directo es frágil: si el fondo cambia, el contraste puede fallar sin que sea evidente.

**Recomendación:** Reemplazar `rgba(255, 255, 255, 0.75)` con un color sólido equivalente (aproximadamente `#BFC4CC`) como variable SCSS explícita.

**ALERTA:** `rgba(255, 255, 255, 0.6)` (username en sidebar) sobre `#122640`. Ratio efectivo: ~5.6:1 — justo por encima del mínimo 4.5:1 para texto caption. Pasa, pero está en el límite. Si el texto de username es caption (12px), el mínimo para texto pequeño es 4.5:1 — pasa. Pero sin margen de seguridad.

### 4.2 Foco visible

**Resultado: DELEGADO A MUI (correcto pero sin customización).**

No hay estilos de focus personalizados en ningún SCSS Module. El outline de foco viene del tema MUI default y de CssBaseline. Los botones, campos y nav items tendrán el focus ring default de MUI (generalmente correcto), pero no se verificó si hay casos donde el outline se pierde contra el fondo marino del sidebar.

**Riesgo específico:** Los `ListItemButton` en `NavSidebar` sobre fondo `#122640` oscuro. Si el focus ring de MUI es oscuro (como en algunos temas), puede volverse invisible. Recomendación: agregar `outline: 2px solid $secondary-main` en el estado focus de los nav items.

### 4.3 ARIA labels

**Resultado: BUENO en los casos existentes.**

- `ConnectivityIndicator`: `role="status"`, `aria-live="polite"`, `aria-label` en el componente y en íconos. Excelente.
- `AppHeader`: `aria-label="abrir navegación"` en el botón de menú. Correcto.
- `OfflineMutationDialog`: `aria-labelledby="offline-mutation-dialog-title"`. Correcto.
- `PasswordTextField`: El botón de mostrar/ocultar contraseña no tiene `aria-label`. El ícono cambia visualmente pero los lectores de pantalla no sabrán si muestra u oculta.

**Recomendación:** Agregar `aria-label={showPassword ? "Ocultar contraseña" : "Mostrar contraseña"}` al `IconButton` en `PasswordTextField`.

### 4.4 Targets táctiles

**Resultado: GENERALMENTE CORRECTO.**

- `Button size="large"`: 48px altura — correcto (>44px mínimo).
- `IconButton` (header): 48px por defecto MUI — correcto.
- `ListItemButton` (nav): altura suficiente por padding — correcto.
- `PasswordTextField` toggle button: 40px default de MUI — debería ser 44px en mobile. Menor.

### 4.5 Jerarquía tipográfica

**Resultado: GAP.**

`ProfilePage` usa `variant="h5"` para el título de página SIN pasar por `ModuleLayout`. El resultado es una jerarquía diferente para esta página vs. las demás (que usan `ModuleLayout` con `h5` también, pero con `fontWeight: 700` forzado en el SCSS). No hay un H1 semántico claro en ninguna página.

`AuthModal` usa `variant="h5"` para "Eventos Hersa" como título del login — esto está bien semánticamente (es el único heading de la pantalla).

**Preocupación principal:** MUI renderiza `<Typography variant="h5">` como `<h2>` por defecto si no se especifica `component`. En `NavSidebar`, el texto "EVENTOS HERSA" usa `variant="h6"` — eso renderiza como `<h6>` dentro de un `<nav>`. La jerarquía semántica H1→H2→H3 no está siendo controlada explícitamente.

## 5. Escalabilidad UI

### 5.1 Qué funciona bien al escalar

- **Design tokens centralizados** (`theme.ts` + `_variables.scss`): el riesgo de color/spacing drift es bajo.
- **`ModuleLayout`**: patrón correcto para páginas nuevas — título, acciones, footer ya resueltos.
- **SCSS Modules** con co-location: no habrá conflictos de selectores entre módulos.
- **Estructura de módulos** (`features/` anidados): permite crecer sin colisiones.
- **`MutationButton`**: el manejo offline no requiere trabajo adicional por módulo.

### 5.2 Qué se romperá primero (al agregar 5–10 módulos)

**Riesgo ALTO — Sin componente de tabla compartido:**
El módulo `admin` actual usa listas. Los módulos futuros (gestión de colegios, graduaciones, pedidos, logística) necesitarán DataGrid. Cada módulo implementará su propia tabla sin convenciones de: header marino, filas alternas, números a la derecha, sort/filter, estados vacío/error, paginación. El drift visual en tablas será el primer punto de inconsistencia visible.

**Riesgo ALTO — Sin componente `EmptyState`:**
Cuando una lista o tabla no tiene datos, cada módulo mostrará algo diferente. En la app actual esto no es evidente porque todo es placeholder. Al conectar datos reales, el problema aparecerá en todas las tablas y listas simultáneamente.

**Riesgo MEDIO — Sin estados de loading de página:**
`ModuleGuard` retorna `null` durante la carga de permisos — pantalla en blanco. Cuando haya más módulos con carga de datos inicial, cada uno lo resolverá ad-hoc (algunos con `Skeleton`, algunos con spinner, algunos con nada).

**Riesgo MEDIO — Dos variantes de formulario de auth:**
`AuthModal` (patrón propio) vs `ForgotPasswordPage`/`ResetPasswordPage` (usan `AuthPageCard`). Si se agrega una pantalla de registro, ¿cuál patrón sigue?

**Riesgo MEDIO — `ComingSoonGrid` con íconos genéricos:**
Cuando se implementen los módulos reales, las páginas placeholder serán reemplazadas, pero si el equipo mantiene `ComingSoonGrid` para nuevas funcionalidades en desarrollo, el patrón de "mismo ícono para todos" continuará.

**Riesgo BAJO — Ausencia de Breadcrumb:**
Los módulos actuales son todos de un nivel. Al agregar sub-módulos (e.g., `grados/ceremonia/123`), el usuario pierde contexto de navegación.

**Riesgo BAJO — `h5`/`h6` sin tokens en tema:**
Cuando se construyan más pantallas, los desarrolladores usarán `variant="h5"` o `variant="h6"` con el default MUI en lugar de los tokens Hersa.

### 5.3 Análisis de `StatCard` — módulo admin

`StatCard` es el componente más elaborado de un módulo. Tiene sus propios estilos para chips de tendencia fuera del sistema semántico de MUI (`className={trendUp ? styles.chipUp : styles.chipDown}` en lugar de `color="success|error"`). Esto es correcto porque los Chips de tendencia no deben ser `Chip` con `color="success"` (que aplica texto claro sobre fondo verde — incorrecto visualmente aquí), pero la implementación es una desviación razonada del sistema.

**Recomendación:** Documentar en el componente por qué no usa `color="success|error"` — de lo contrario, revisiones futuras eliminarán los estilos SCSS pensando que son redundantes con el prop semántico.

## 6. Cumplimiento SCSS Modules

### 6.1 Evaluación general

**Resultado: EXCELENTE.**

De todos los componentes y páginas auditadas, ninguno usa `sx` prop ni `style={{}}` inline. El patrón SCSS Module con co-location se respeta al 100% en:
- Todos los componentes en `shared/components/`
- `AuthModal`
- `AdminPage`, `GradosPage`, `TiendaPage`
- `ProfilePage`, `ForgotPasswordPage`, `ResetPasswordPage`
- `ChangePasswordForm`, `ForgotPasswordForm`, `ResetPasswordForm`
- `StatCard`

### 6.2 Hallazgo único

El único `style={{}}` encontrado es en `Layout.tsx`:
```tsx
const mainStyle =
  indicatorHeight > 0
    ? ({ "--indicator-offset": `${indicatorHeight}px` } as React.CSSProperties)
    : undefined;
// ...
<Box component="main" className={styles.main} style={mainStyle}>
```

Esto es una excepción justificada: pasa una CSS custom property calculada dinámicamente que no puede expresarse en un SCSS Module estático. Es el patrón correcto para este caso — no es una violación.

**Conclusión del punto 6:** El equipo ha internalizado correctamente la convención. El riesgo de drift hacia `sx` es bajo basado en los patrones actuales.

## 8. Recomendaciones Priorizadas

### Prioridad 1 — Alto impacto, bajo esfuerzo

**R1. Unificar el patrón de pantallas de autenticación.**
Refactorizar `AuthModal` para usar `AuthPageCard` en lugar de su propio `Paper` + `Container`. Elimina la duplicación de patrón de auth y consolida un solo punto de cambio para el diseño de pantallas de autenticación.
- Archivos: `AuthModal.tsx`, `AuthModal.module.scss`
- Esfuerzo: ~1–2 horas

**R2. Agregar hover state a tarjetas (Paper/Card outlined).**
Implementar el hover del brand (`border-color: $primary-light`) globalmente via override en `theme.ts` o como mixin SCSS compartido.
- Archivos: `theme.ts` (override `MuiPaper`/`MuiCard`) o `_variables.scss` + mixin
- Esfuerzo: ~1 hora

**R3. Agregar `aria-label` al toggle de contraseña en `PasswordTextField`.**
Fixes a11y real: lectores de pantalla no anuncian el estado actual del toggle.
- Archivos: `PasswordTextField.tsx`
- Esfuerzo: ~15 minutos

### Prioridad 2 — Alto impacto, esfuerzo medio

**R4. Crear componente `EmptyState`.**
Componente compartido con: ícono (configurable), título, subtítulo opcional, acción opcional (botón). Evita que cada tabla/lista implemente su propio estado vacío. Consumible por cualquier módulo futuro.
- Ubicación: `shared/components/EmptyState.tsx`
- Esfuerzo: ~3–4 horas

**R5. Agregar overrides de MUI al tema para Card/Paper.**
Agregar en `theme.ts` bajo `components`:
```ts
MuiPaper: { defaultProps: { elevation: 0 } },
MuiCard: { defaultProps: { variant: "outlined" } },
```
